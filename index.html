Server-side compaction

How LLM agents keep running without hitting the context wall

The problem: context windows fill up

Every LLM operates within a finite context window — the total number of tokens it can see at once. Short conversations never stress this limit. But LLM agents are different: they read files, call tools, reason through results, and iterate — sometimes for hours. Each step burns tokens.

Finite window sizes

Claude 3.x: 200K · GPT-4o: 128K · Gemini 1.5: up to 1M tokens. Large, but not infinite.

Agents are token-hungry

A single file read: 2–5K tokens. Compiler output, test logs, reasoning chains — it accumulates fast.

The failure mode

At the limit the agent errors out, loses early context, or forces manual intervention — breaking automation.

Naive solutions — and why they fail

Three approaches look obvious. All share a fatal flaw: they destroy information the agent needs, or they break automation entirely.

Modern compaction techniques

Four techniques represent the current state of the art. They operate at different layers of the stack — application, token, model, and infrastructure — and make different tradeoffs between information fidelity, latency, and implementation complexity.

① Hierarchical (tiered) summarization

The most widely deployed technique in production agents. Instead of collapsing everything into one flat summary, hierarchical compaction maintains multiple memory tiers at different levels of detail. Recent context stays verbatim; older context is progressively compressed into episode summaries and then into extracted key facts. Each tier is regenerated incrementally — only the new chunk is summarized and folded into the existing tier, so there's no expensive full re-summarization on every step.

Used by Claude Code, LangChain's ConversationSummaryBufferMemory, and LlamaIndex's chat memory. The trigger is typically token budget: "when the window reaches 70–80%, summarize the oldest chunk and fold it into Tier 2."

Tier 1 — verbatim

Last N turns kept exactly. Full fidelity — exact error messages, line numbers, tool outputs.

Tier 2 — episodes

Older turns compressed to per-phase summaries. "Explored auth module, found 3 issues, fixed 2."

Tier 3 — facts

Key decisions, goals, constraints extracted. Always injected. Survives indefinitely.

Advantages

  • Recent context stays high-fidelity
  • Goals and constraints always present
  • Incremental — no expensive full re-summarize
  • Production-proven in Claude Code, LangChain

Limitations

  • Tier boundaries need tuning per use case
  • Summary quality is model-dependent
  • Still lossy — details vanish over time

② Token-level pruning (LLMLingua / Selective retention)

Rather than summarizing at the message level, token-level pruning operates at the individual token or sentence level. Each token or sentence is assigned an importance score — typically derived from a smaller "compressor" model's perplexity signal (low perplexity = predictable = low information = pruneable). High-scoring tokens are kept verbatim; low-scoring ones are dropped. The result is a compressed prompt that preserves the semantic content of the original with minimal structural change.

LLMLingua (Microsoft, 2023) and its successor LongLLMLingua achieve 3–20× compression ratios with less than 5% performance drop on many benchmarks. A separate approach — importance-scored selective retention — works at the message level: assign each turn a role-based score (goal-setting turns score high, filler turns score low) and discard below the threshold.

Advantages

  • Fine-grained — preserves high-signal tokens
  • Works at application layer, no model access needed
  • 3–20× compression with small quality drop
  • Compositional — can combine with summarization

Limitations

  • Requires a compressor model (extra inference cost)
  • Perplexity ≠ task relevance in all cases
  • Disrupts token order in unpredictable ways
  • Heuristic scoring can misclassify critical content

③ KV cache eviction (H2O / SnapKV / StreamingLLM)

This technique operates at the model inference layer, not the application layer. During inference, transformers maintain a Key-Value (KV) cache — a compressed representation of all past tokens used by the attention mechanism. These methods shrink the cache by evicting low-attention tokens, giving the model an effectively longer context than its nominal window.

  • H2O (Heavy-Hitter Oracle) — identifies tokens that accumulate the most cumulative attention mass ("heavy hitters") across heads and layers, evicting the rest. Achieves ~4× compression.
  • SnapKV — clusters similar key vectors per attention head and selects representative tokens. Significantly outperforms H2O on long-context benchmarks.
  • StreamingLLM — always preserves "attention sink" tokens (usually the first 4 tokens) plus a recent sliding window, enabling infinite-length generation without quality collapse.

Advantages

  • Transparent to the application layer
  • No extra inference calls needed
  • Attention sinks preserve coherence
  • H2O achieves ~4× compression; SnapKV exceeds it

Limitations

  • Requires model-level access — not available via API
  • Attention ≠ semantic importance in all cases
  • Behavior varies significantly across architectures
  • Still experimental outside research settings

④ Virtual paging / retrieval-augmented memory (MemGPT-style)

Inspired by OS virtual memory, this approach treats the context window as a page in RAM and an external vector database as disk. Old context is serialized and stored externally. When the agent needs information from its history, it retrieves the relevant passages and pages them back in. The agent itself decides when to page out and what to retrieve — typically via function calls to a memory API.

MemGPT (Packer et al., 2023) pioneered this pattern. Production implementations include Zep, Mem0, and various LangGraph memory modules. The key insight: no information is ever permanently deleted — it's just moved to cheaper storage.

Advantages

  • Theoretically unlimited memory
  • Nothing permanently deleted — only moved
  • Agent retrieves only what's relevant now
  • Composable with summarization at page-out time

Limitations

  • Retrieval can miss critical context
  • Adds latency on every memory lookup
  • Requires vector DB infrastructure
  • Agent must be trained to use its own memory

Comparison

Technique Layer Info fidelity Complexity Best for
Hard truncation naive Application Recent only — early context lost Trivial Short tasks, disposable early context
Sliding window naive Application Recent only — system prompt can scroll off Trivial Rarely appropriate for agents
Flat summarization basic Application High-level summary only Low Moderate-length runs, simple tasks
Hierarchical summarization modern Application Multi-resolution — recent verbatim + summaries Medium Long agent runs, complex multi-step tasks
Token-level pruning modern Application / prompt Selective — high-signal tokens preserved Medium Prompt compression, dense information retrieval
KV cache eviction model-level Model inference Attention-weighted — heavy hitters preserved High (needs model access) Research, self-hosted models, inference optimization
Virtual paging infrastructure Infrastructure Near-lossless — everything stored externally High Very long runs, knowledge-intensive agents

Universal tradeoffs

Every compaction technique navigates the same three-way tension. There is no free lunch.

Fidelity ↔ Size

Keeping more information costs more tokens. You cannot compress without some loss. The question is always which information to sacrifice.

Latency ↔ Quality

Better summaries require more inference time. Fast compaction (truncation) is instant but lossy. Careful summarization takes seconds.

Complexity ↔ Control

Sophisticated techniques (paging, KV eviction) give better results but are harder to implement, debug, and reason about in production.

The practical recommendation for most production agents today: start with hierarchical summarization — it's well-understood, API-compatible, and production-proven. Layer in token-level pruning for high-density prompts. Consider virtual paging only when tasks genuinely span thousands of steps or require recalling arbitrary past events.
